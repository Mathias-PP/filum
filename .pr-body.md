## Résumé

Trois fixes indépendants :

1. **Boutons header overflow sur mobile** — "Se connecter" et "Créer une fiche" se cassaient sur 2 lignes au lieu d'être centrés / sur une ligne.
2. **Auth Google échoue sur mobile** (PC OK) — cookie de session bloqué par ITP/Safari quand frontend et backend sont sur des origines différentes.
3. **Wayback Machine pas visible** — service auto-archive ne déclenchait jamais vraiment l'archivage, et pas de moyen manuel.

---

## 1) Boutons mobile (`Button.svelte` + `+layout.svelte` + `tailwind.config.js`)

- Ajout de `whitespace-nowrap` aux classes de base de `<Button>`. Garde le texte sur une ligne quoi qu'il arrive, plus de retour à la ligne qui défonce la `h-8`. Bénéficie à tous les boutons du site, pas seulement ceux du header.
- Sur les téléphones les plus étroits (< 480 px), le bouton "Créer une fiche" est masqué (`hidden xs:inline-flex`) — il pointe vers la même URL Google que "Se connecter", donc aucune perte fonctionnelle. Sur tablette et desktop les deux restent visibles.
- Nouveau breakpoint Tailwind `xs: 480px` ajouté dans `tailwind.config.js` (intercalé entre 0 et le `sm: 640px` standard).

## 2) Auth Google sur mobile (`apps/frontend/vercel.json`)

**Diagnostic** : sur mobile (Safari iOS, Chrome iOS qui partage WebKit, et l'ITP de Safari sur macOS aussi), les cookies "tiers" sont bloqués par défaut. Quand le frontend (`filum.app` ou `filum.vercel.app`) appelle le backend (`*.up.railway.app`), le cookie `filum_session` posé par Railway est considéré comme tiers même avec `SameSite=None; Secure` (déjà bien configuré côté backend). Résultat sur le `/auth/callback` : le `GET /api/v1/auth/me` n'envoie pas le cookie → 401 → message "Échec de l'authentification. Redirection…".

**Fix** : on rend l'API first-party via une réécriture Vercel. `apps/frontend/vercel.json` proxy `/api/*` vers le backend Railway. Le frontend appelle `/api/v1/...` (chemin relatif, ce qui est déjà le fallback quand `PUBLIC_API_BASE_URL` est absent), Vercel route vers Railway côté serveur, le navigateur ne voit qu'une seule origine, le cookie est first-party.

**⚠️ Étapes à faire après le merge** (peuvent pas être automatisées depuis le code) :

1. **Éditer `apps/frontend/vercel.json`** : remplacer `REPLACE_WITH_BACKEND_HOST` par le vrai hostname Railway (ex. `filum-backend-production.up.railway.app`). Je n'ai pas l'URL exacte donc j'ai laissé un placeholder explicite.
2. **Sur Vercel → Project Settings → Environment Variables** : supprimer (ou vider) `PUBLIC_API_BASE_URL`. Le client fallback déjà vers le chemin relatif quand cette var est vide.
3. **Redéployer**.

Une fois ces deux étapes faites, le mobile devrait s'authentifier exactement comme le desktop.

## 3) Wayback Machine — auto-archive qui marche + champ manuel

**Diagnostic** : `WaybackService.archive_url` ne faisait qu'interroger `archive.org/wayback/available` — qui retourne le snapshot **existant le plus proche**, sans en créer. Pour des URLs récentes (typique du contenu démo / blog post / article frais) il n'y a pas de snapshot pré-existant, donc le service marquait toujours `FAILED`. D'où l'absence de `archive_url` sur les fiches malgré le service présent.

**Fix backend** :

- `app/services/wayback.py` refait : déclenche d'abord un **Save Page Now** (`GET https://web.archive.org/save/<url>`, best-effort, on swallow les erreurs), puis poll `wayback/available` avec back-offs croissants (3 s → 5 s → 8 s → 8 s → 9 s, ~33 s total). Quand un snapshot apparaît, on met `archive_status=ARCHIVED` + `archive_url` + `archive_timestamp`. Sinon `FAILED`. Tout ça reste dans le `asyncio.create_task` background donc ça ne bloque pas la réponse à l'utilisateur.
- `app/schemas/source.py` : `SourceCreate` et `SourceUpdate` acceptent maintenant un champ optionnel `archive_url`. Quand fourni, le backend l'utilise tel quel et **skip le background task** (status `ARCHIVED` immédiat).
- `app/api/v1/endpoints/sources.py` : POST et PATCH gèrent le `archive_url` manuel. Vider la valeur via PATCH remet le status à `PENDING`.

**Fix frontend** :

- `apps/frontend/src/lib/api/types.ts` : `SourceCreate` ajoute `archive_url?: string | null`.
- `apps/frontend/src/routes/dashboard/new/[card_id]/sources/+page.svelte` : nouveau champ "Lien archivé (optionnel)" dans le formulaire, avec un placeholder Wayback et un texte d'aide expliquant le fallback automatique.
- L'affichage de `archive_url` existait déjà sur la fiche publique et `SourceDetailPanel` — pas besoin d'y toucher, ça commencera juste à s'afficher dès qu'une source aura un `archive_url` non null.

**Côté démo `/@example/memoire-et-cerveau`** : les sources existantes n'ont rien parce qu'elles ont été créées avec l'ancienne version cassée du service. Elles peuvent être éditées une à une via le dashboard (le champ accepte maintenant un Wayback URL manuel, ou laissé vide pour relancer l'auto-archive après que le user re-publie).
